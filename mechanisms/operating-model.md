# The Operating Model — How the Pieces Compose Into a Running Loop

## What it is

The operating model is the runtime assembly of every other mechanism: the shape the studio actually takes on a working day, and the path a unit of work travels from request to merge. Its one axiom: **the studio is one director session the human talks to, a small set of disposable team sessions doing the work, and a message bus carrying state between them — and the human's review bandwidth is the clock that caps how many teams run at once.** Everything else in this repo is a part; this is how the parts turn. The director is the human's single point of contact — the human mostly talks only to it — and each team is a separate session owning one lane of work. Between sessions, state moves on a message bus that is either the repository (when sessions run automated and read their own state) or the human (when the human copy-pastes prompts and outputs between a desktop director session and a desktop team session). Both are legitimate; they trade autonomy for the human's eyes-on, and a project can run in either mode — or both at once — without changing any of the underlying mechanisms.

## The protocol

**The session topology.**

1. **One director/orchestrator session** is the human's single communication layer. It does not, as a rule, write feature code; it reviews what the teams land, verifies it adversarially, merges what is in its lane, escalates design law to the human as ballots, and keeps every tracking surface true. Exactly one director is live at a time (the one-orchestrator-at-a-time guard, see `boot-shutdown.md`); if a fresher dispatch than your own boot exists, another director is running — stop and reconcile before writing.
2. **Each team is its own session,** dedicated to one feature area or kind of work — e.g. a combat team, a lore/narrative team, an economy team — each with a narrower lane, reading its own charter, workplan, and the rulings in scope. A team session never sees another team's session; its boundary is its charter, not a branch.
3. **The human sits above the director,** spending attention only on the reserved lane: feature merges, design ballots, hands-on/in-use verification, and taste.

**The daily loop (what one working session actually runs).**

1. **Boot.** Every session — director and team — reconstructs state from the repo in the fixed order and runs its self-checks (`boot-shutdown.md`). The director's confirm-read is a one-paragraph state-of-the-world; a team's is its lane's state.
2. **Work the queue.** The director surfaces anything needing the human as ballots (`ballots.md`) and keeps in-lane work moving. Teams draft against their workplans, attaching a Definition-of-Done rung and authentic evidence to each unit (`done-ladder.md`).
3. **Review and gate.** Work enters the review pipeline: author self-check + evidence → lead/diff review → adversarial verification (the load-bearing stage) → automated gates → merge (`review-pipeline.md`). Verification outranks generation — the strongest configuration is spent on checking, not producing.
4. **Merge by lane.** Green-lane, reversible classes (docs, plans, memory, logbook) merge on the director's signatures; reserved classes (code, assets, taste-bearing files) go to the human as a feature decision, unless a standing delegation covers them (`lanes-and-autonomy.md`).
5. **Shutdown.** Before any session ends, commit and push everything, re-stamp the roadmap/status, write the dispatch, update memory pointers, and leave the board current (`boot-shutdown.md`). An uncommitted artifact does not exist to the successor.
6. **Measure.** Each gate verdict is captured into the review ledger as it is recorded, and the director reads the dashboard at boot (`measurement/`). The ledger is the quantitative feed; the human-facing status surface is the qualitative one.

*Concretely, following one unit through the loop:* the **combat** team session drafts a new combat-state behavior against its workplan and attaches its evidence and target Definition-of-Done rung (step 2); the **director** picks it up, verifies it adversarially rather than trusting the draft, and runs the automated gates (step 3); because the change is reserved-lane code, not green-lane docs, the director does not self-merge it (step 4) but surfaces it to the **human** as a feature ballot; the human's ratification comes back across the bus and is recorded verbatim before the merge executes. A lore-team wording change or an economy-team memory note would travel the same path but land in the green lane, merging on the director's signatures without a ballot.

**The two message-bus modes (name the tradeoff, pick per situation).**

- **Repo-as-message-bus (automated).** In cloud/CLI runs, sessions communicate only through durable repo state: a team writes its output and its dispatch, and the director's next boot reads it. No human is in the loop between sessions. Highest autonomy, lowest touch — and it works *only because* boot/shutdown are clean (state lives in the repo, never in the model). This is the mode the boot-prompt template and the whole statefulness protocol are built for.
- **Human-as-message-bus (hands-on desktop).** In the common desktop workflow, the director session and each team session run in separate chat surfaces that cannot see each other, so the *human* carries state between them by copy-paste: paste the director's ask into the team session, paste the team's output back into the director. Higher-touch and slower, but it gives the human eyes on every message that crosses — which is often exactly what the human wants for weighted or early-stage work. Neither mode is more correct; choose by how much the work needs the human's eyes versus its throughput.

**The copy-paste relay protocol (how the human bus keeps from dropping context).** When the human is the bus, a relayed prompt must be self-sufficient, because the receiving session shares no memory with the sender. Carry three things, every hop:

1. **The boot prompt** — so the team session reconstructs its lane's state from the repo before acting (never a bare ask into a cold session). This is the parameterized team boot prompt, resolved.
2. **The specific ask** — the one unit of work, its target Definition-of-Done rung, and the exact gate list it must pass. One relayed message should carry one decidable unit, not a discussion.
3. **The return format** — the structured-output shape the director will gate on (findings / gate-results / merge-recommendation), so the team's reply arrives as something to gate, not prose to re-parse. The relay payload *is* the agent briefing kit, delivered by hand instead of by harness. When the reply comes back, the director records any human ratification verbatim (`ballots.md`) before executing on it.

**Team-count discipline (the human is the clock).** The number of *active* teams is capped by the human's review bandwidth, not by how much work exists — every team's output eventually lands on the human's desk as a decision, so more teams than the human can review just grows an unreviewed backlog that looks like progress. Run a small steady-state set (on the order of two), burst to a few more only briefly, and hold the rest dormant. A team is "active" when it is producing work the human must gate; parking a team is free, over-running the human is not. When adding a team would outrun the human's ability to review, the answer is a ballot, not a new session.

## Why it exists

- The single-director-as-communication-layer topology is the studio's actual shape: one orchestrating session runs the project between the human's touches, reviews what teams land, gates adversarially, and escalates as ballots — and only one runs at a time (`studio:docs/kb/director-handbook.md` §2, §10; the one-orchestrator guard in `studio:docs/kb/studio-operations.md`).
- Each team being its own session with a narrower lane, booting from its own charter/workplan/rulings, is the team-session design carried in the parameterized generator and the operations doc (`bindings/boot-prompt-template.md`, the team-session prompt; `studio:docs/kb/studio-operations.md`, the per-team session-end ritual added after "the first two team sessions each missed part of it").
- Team-count-capped-by-human-bandwidth is explicit standing law: the active set is capped at "two teams steady-state, three burst," because "the human's review bandwidth is the studio clock" (`studio:CLAUDE.md`, "The teams"; `bindings/example-studio.md` §team roster).
- Repo-as-message-bus is the founding axiom of statefulness — "state lives in the repo, never in the model," so a disposable session's successor resumes purely from durable surfaces (`studio:docs/kb/director-handbook.md` §2; `mechanisms/boot-shutdown.md`). Earned independently as the validation project's end-of-session discipline: everything the successor needs is committed before the session ends, or "an uncommitted artifact does not exist to the successor" (`validation:.claude/skills/end-session/SKILL.md`).
- Human-as-message-bus is the real desktop workflow: the pasteable boot prompt exists precisely to be pasted by the human into a fresh session (`studio:docs/kb/director-handbook.md` §10), and the desktop surfaces are siloed — the code surface and the desktop-chat surface read *separate* tool configs, so a session cannot see another's state and the human bridges them (`studio:CLAUDE.md`, the MCP-config section; `validation:CLAUDE.md`, the two-config gotcha). The validation project's whole cadence is a human relaying between a drafting agent and the ground truth — "the agent drafts; the human reviews and tests — the review/test loop is the actual quality gate" (`validation:CLAUDE.md`, operating style).
- The copy-paste relay payload is the agent briefing kit: every agent prompt already carries the state read, the specific ask + its gate list, and the required structured-output schema, so nothing arrives as prose to re-parse (`bindings/example-studio.md` §8; `studio:docs/kb/director-handbook.md` §5). Delivering it by hand instead of by harness changes the transport, not the contract.
- The measurement half — the review ledger fed at gate time and read as a boot dashboard — already composes into the boot ritual without a manual step (`measurement/boot-integration.md`, Stage 0/1; `mechanisms/review-pipeline.md`).
- Two-repo test: the disposable-session / state-in-the-repo topology and its boot-shutdown handshake appear independently in the studio project (director + team sessions, one-orchestrator guard) and the validation project (end-session skill, commit-before-successor); the human-in-the-loop relay appears independently in the desktop-driven studio workflow (pasteable prompts across siloed surfaces) and the validation project (draft→human→test loop).

## Failure modes of the mechanism itself

- **Too many teams for the human.** Spinning up more active teams than the human can review, so output piles unreviewed and the backlog masquerades as velocity. The cap exists because the human is the clock; when a new team would outrun review, that is a ballot, not a session.
- **Relay context drop.** Pasting a bare ask into a cold team session with no boot prompt, or pasting a reply back with no return-format structure — the receiving session shares no memory, so it acts on a false or partial world. Every hop carries all three: boot prompt, specific ask, return format.
- **Two directors live.** Running a second orchestrator session (easy to do by accident across two desktop windows) so two directors clobber each other's state. The fresher-dispatch check is the guard; one director at a time is absolute.
- **Mode confusion.** Assuming repo-as-bus autonomy while actually running siloed desktop sessions the human must relay — the "other session will just read it" that never happens because nothing was pushed, or because the sessions can't see the same surface. Name which bus is in force before relying on either.
- **Director drafting instead of gating.** The director quietly authoring feature work rather than reviewing it collapses the author/gate separation the whole pipeline rests on (`review-pipeline.md`). The director's lane is gating and escalation; feature generation is the teams'.
- **Relay as decision laundering.** Treating a copy-pasted "ok" as a ratified decision without recording it verbatim, so a later session can't tell a real ratification from an invention. Every human decision that crosses the bus is recorded verbatim before it is executed on (`ballots.md`).

## What varies per project

The bindings layer declares **which sessions exist** and how they map to feature areas (the team roster and codenames), the **active-team cap** for this project's human bandwidth, **which message-bus mode(s)** are in force and the surfaces each session reads/writes, the **relay payload specifics** (the resolved team boot prompt, the standard ask fields, the structured-output schema), and the **queue order** the director works after boot. The *shape* — one director as the human's single layer, teams as siloed per-lane sessions, a message bus that is either the repo or the human, the relay-carries-three-things rule, and the human-bandwidth cap on active teams — is framework law; the roster, cap number, and transport are bindings. See `bindings/TEMPLATE.md` → project identity, autonomy lanes, tracking surfaces, and agent-briefing kit, and `bindings/example-studio.md` for a fully-populated topology (director role + the standing team roster + the two-steady/three-burst cap).
