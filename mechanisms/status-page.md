# The Human Status Page — the one surface the human reads

## What it is

**Status: [PROPOSED].** The status page is a director-maintained, plain-English page that projects *over* the machine tracking surfaces — the board, the roadmap/state file, the ruling register, the measurement ledger — so the human lead never has to read any of those to know where things stand. It rests on one axiom: **the machine surfaces are authoritative, but they are not readable by a human at a glance.** The roadmap tracks state, the register tracks law, the board tracks work, the ledger tracks quality — each true, each in its own format, none of them a single skimmable answer to "what needs me, and where are we?" The status page is the answer: a read-only human consolidation, regenerated by the director every shutdown and read first at every boot, that leads with the decisions and tests awaiting the human and stays skimmable in under a minute. It is a *projection*, never a source — if it disagrees with the surfaces it renders, they win and it gets regenerated.

Why `[PROPOSED]`: its constituent duties are all earned law — the shutdown re-stamp of a status surface, the boot's one-paragraph state-of-the-world, ballots that must never be buried in chat, the boot dashboard, the stranded-work scan. What is *new* is consolidating them into one human-first page. The need was flagged (tracking-surface consolidation) and the human lead named the pain directly — a plain-English, follow-along status page would have spared much of the confusion of reading the machine surfaces to keep up — but that is a feature request, not yet its own dated scar. The page is honest about this: every duty it composes cites a real receipt below; the composition itself is the proposed increment and is labeled so until it earns its own incidents.

## The protocol

**The page is a director duty, not a team artifact.** The orchestrator owns it; teams feed the surfaces it projects over, they do not write it.

**Its spine — six sections, in this priority order (most important first, because the human reads top-down and stops when satisfied):**

1. **Your queue** *(top — what the human reads first).* Two lists: (a) **decisions awaiting the human** — every open ballot, one line each, each with its marked recommendation so a one-word reply settles it; (b) **awaiting the human's test / sign-off** — the L4 items sitting L2-verified and needing his play/use to reach milestone-done. This section is the whole point of the page.
2. **Where we are.** The current milestone/phase in one sentence, plus a one-line health note (trunk green/red, anything blocking).
3. **What each team is doing.** A compact table — team, current task, status (active / blocked / waiting-on-human / idle), last update. One row per team; no prose.
4. **What's next.** The next milestone and its done-criteria, plus any kill/pivot trigger written before the work starts.
5. **Where the work lives.** A live index of in-flight branches / PRs / sessions — the fix for "it's buried in a worktree I can't find." Mirrors the boot's stranded-work scan, surfaced for the human.
6. **Recently landed.** The last few merges and rulings, condensed — the human's changelog since he last looked.

**Update cadence (the load-bearing rule):**

- **Regenerated at every shutdown** as a binding session-end duty, rebuilt from the trackers/roadmap/register/ledger and stamped, in the same push as the roadmap re-stamp (`boot-shutdown.md` shutdown step 2 — the STATUS-regeneration sub-step nested under it).
- **Read first at boot**, then reconciled against `git log` and the trackers in the confirm-read step — because a prior dirty shutdown could have left it stale, the boot never *trusts* it, it *verifies* it; and on the first-ever boot of a freshly-instantiated project — before any shutdown has run — the page may not exist yet, in which case the boot notes its absence and proceeds, the first shutdown creates it (`boot-shutdown.md` boot step 8).
- **The Your-queue section is never allowed to ship stale.** A resolved ballot still listed, or a new decision not surfaced, is the one drift the page exists to prevent; that section must reflect the true open set every single session, even when the rest of the page is only lightly touched.
- **It is regenerated, never hand-edited.** The director rebuilds it from the surfaces; nobody edits it as though it were the source. It is a render.
- **It stays a dashboard, not a log.** Each section is capped (a handful of rows); the changelog is condensed, not exhaustive. If it can't be skimmed in a minute, it has failed.

## Why it exists

- The boot ritual already ends by producing a one-paragraph "state-of-the-world / state-of-the-studio" and surfacing anything needing the human that day (`studio:docs/kb/director-handbook.md` §2; mirrored in `mechanisms/boot-shutdown.md` boot step 8). The status page is the durable materialization of that same paragraph — written down instead of spoken once and lost.
- Shutdown already re-stamps the roadmap/status surface to the session's final state, in the same push, every session (`studio:docs/kb/director-handbook.md` §2b; `mechanisms/boot-shutdown.md` shutdown step 2). The status page extends that earned re-stamp discipline from a machine surface to a human-facing one.
- Open questions to the human must be queued as ballots and **never buried in chat** (`mechanisms/boot-shutdown.md` shutdown step 5; `mechanisms/ballots.md`); when the human is absent multi-day, ballots queue on the board and he returns to a decision queue (`studio:bindings/example-studio.md` §2, availability rhythm). The Your-queue section is that decision queue rendered where he will actually see it, recommendations attached.
- The "where the work lives" problem is a paid-for scar: the boot self-check scans stranded worktrees because a session's entire output once "nearly died" uncommitted in a tree no one could find (`studio:docs/kb/director-handbook.md` §6; `mechanisms/boot-shutdown.md` boot step 6). Section 5 surfaces that same in-flight index to the human so no work is invisible to him.
- The quantitative feed already exists and is already wired into the director boot ritual: the review-ledger dashboard (`measurement/boot-integration.md` Stage 1; `studio:measurement/review-ledger.jsonl`, 34 rows live since 2026-07-08). The status page **composes over** that dashboard for the health/quality note — it does not duplicate it; the ledger stays the quantitative surface, the page is the human-readable one.
- The validation project independently runs the mirror discipline: its end-session ritual re-stamps project state and its boot reads it first (`validation:.claude/skills/end-session/SKILL.md`), the same write-at-shutdown/read-at-boot loop this page binds to.
- **Two-repo test:** the pattern "re-stamp a status surface at shutdown, read it first at boot" is present independently in the studio project (roadmap re-stamp + state-of-the-studio confirm-read) and the validation project (end-session state re-stamp + boot read). The `[PROPOSED]` increment — folding those into a single plain-English, human-first page whose top section is the pending-decision + awaiting-test queue — is what awaits its own dated receipts.

## Failure modes of the mechanism itself

- **Stale page.** Not regenerated at shutdown, so the human boots into a false world and decides against where things stood a session — or a week — ago. This is the exact confusion the page exists to end; it is prevented only by binding the refresh to shutdown step 2 (its STATUS-regeneration sub-step), never to "when I get to it."
- **Stale Your-queue (the worst variant).** A ballot resolved but still listed, or a fresh decision not surfaced, in the one section the human reads first. Staleness here is worse than anywhere else on the page — it either wastes his attention on a settled question or hides one that's blocking work. This section must be true every session even when nothing else changed.
- **Data-dump bloat.** The page grows into a full changelog / activity log, stops being skimmable in a minute, and the human stops reading it — back to reading machine surfaces, which is the failure it was built to remove. Cap every section; it is a dashboard, not a record.
- **Divergence by hand-editing.** Someone edits the page directly instead of regenerating it, and it drifts from the roadmap/board/register it is supposed to render. It is a projection; the moment it is treated as a source it can lie. Always regenerate from the surfaces.
- **Substitution for the machine surfaces.** Treating the dashboard as the source of truth and letting the roadmap/register/board rot beneath it. The page is a read-only human render; the machine surfaces remain authoritative (the trust order of `mechanisms/memory-trust-order.md` applies — the page is a pointer, the surfaces are ground truth).

## What varies per project

The bindings layer names the **path** of the status page (root `STATUS.md` by convention), its **regeneration source** (which roadmap, board, ruling register, and measurement ledger it projects over), the **section set** if a project needs fewer or renames them, and the **cadence binding** (every shutdown; read first at boot). The *shape* — the six-section priority spine, Your-queue on top, regenerate-don't-edit, skimmable-in-a-minute, never-stale-queue — is framework law; the paths and sources are bindings. At project setup the root `STATUS.md` is seeded once from `bindings/STATUS-template.md` as an explicit day-one instantiation step, so the very first boot's status-page read resolves to a real file; every shutdown thereafter regenerates it, and a genuinely absent page (a first boot before the first shutdown) is tolerated by the boot's note-and-proceed clause. The page is conceptually the human-facing render of the section-6 tracking surfaces plus the roadmap state file plus the ledger dashboard, so it is declared alongside them. See `bindings/TEMPLATE.md` §6 (tracking surfaces), `bindings/example-studio.md` §6 for the populated bindings, `bindings/STATUS-template.md` for the drop-in skeleton, and `bindings/example-status.md` for a worked page.
